# 诊断

一套应对棘手 Bug 的方法论。除非有明确理由，否则不要跳过任何阶段。

**核心原则：** 在尝试修复之前，务必先找到根本原因。只修症状就是失败。

## 铁律

```
不做根因调查，不许提修复方案
```

如果还没完成根因调查，就不能提出修复方案。

## 六个阶段

### 阶段 1 — 建立反馈回路

**这就是核心技能。** 其他一切只是机械操作。如果你拥有一个快速、确定、可由 agent 运行的通过/失败信号，你就能找到原因——二分查找、假设验证和探测都只是消耗这个信号。如果你没有这样的信号，再怎么看代码也无济于事。

在这里投入不成比例的努力。**要积极，要有创意，绝不放弃。**

#### 构建反馈回路的方法——按大致顺序尝试

1. 在能触及 Bug 的任何接缝处编写**失败的测试**——单元测试、集成测试、端到端测试。
2. 对正在运行的开发服务器发起 **Curl / HTTP 脚本**请求。
3. 使用固定输入的 **CLI 调用**，将 stdout 与已知的正确快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，对 DOM/控制台/网络进行断言。
5. **重放捕获的 trace。** 将真实的网络请求/负载/事件日志保存到磁盘；在隔离环境中通过代码路径重放。
6. **临时测试架。** 启动系统的最小子集（一个服务，模拟依赖），用单个函数调用来触发 Bug 代码路径。
7. **属性/模糊测试循环。** 如果 Bug 是"有时输出错误"，运行 1000 个随机输入并查找失败模式。
8. **二分测试架。** 如果 Bug 出现在两个已知状态之间（提交、数据集、版本），自动化"在状态 X 启动，检查，重复"，以便你可以 `git bisect run`。
9. **差异循环。** 在旧版本与新版本（或两种配置）中运行相同输入并 diff 输出。
10. **HITL bash 脚本。** 最后的手段。如果必须由人类点击，用 `scripts/hitl-loop.template.sh` 驱动他们，这样循环仍然是有结构的。捕获的输出反馈给你。

#### 迭代优化回路本身

把反馈回路当作一个产品来对待。一旦有了*一个*回路，问自己：

- 能让它更快吗？（缓存设置、跳过无关初始化、缩小测试范围。）
- 能让信号更清晰吗？（对特定症状做断言，而不是"没崩溃"。）
- 能让它更确定吗？（固定时间、种子随机数、隔离文件系统、冻结网络。）

一个 30 秒的 flaky 回路几乎等于没有回路。一个 2 秒的确定性回路是调试的超能力。

#### 非确定性 Bug

目标不是一个干净的复现，而是**提高复现率**。循环触发 100 次，并行化，增加压力，缩小时间窗口，注入睡眠。50% 概率出现的 Bug 是可调试的；1% 则不行——持续提高复现率直到可调试。

#### 当你确实无法建立反馈回路时

停下来，明确说明这一点。列出你尝试过什么。向用户请求：(a) 能复现该问题的环境访问权限，(b) 捕获的工件（HAR 文件、日志转储、核心转储、带时间戳的屏幕录制），或 (c) 添加临时生产环境探测的权限。在没有回路的情况下**不要**进入假设阶段。

在拥有你信任的回路之前，不要进入阶段 2。

### 阶段 2 — 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们往往直接包含解决方案
   - 完整阅读堆栈跟踪，记下行号、文件路径、错误码

2. **稳定复现**
   - 你能可靠地触发它吗？具体的复现步骤是什么？
   - 每次都能复现吗？如果无法复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 什么变更可能导致了这个问题？
   - `git diff`、最近的提交、新依赖、配置变更、环境差异

4. **在多组件系统中收集证据**
   - 对每个组件边界：记录进入/离开的数据，验证环境/配置的传递，检查每一层的状态
   - 执行一次以收集证据，确定断裂点在哪里，然后针对故障组件深入调查

5. **跟踪数据流**（参见本目录下的 `root-cause-tracing.md`）
   - 错误值从哪里产生的？谁用错误值调用了这里？
   - 持续向上追踪直到找到源头
   - 在源头修复，而不是在症状处修复

### 阶段 3 — 模式分析

**先找到模式，再修复：**

1. **找到可正常工作的示例**——在同一代码库中找到类似的正常代码
2. **与参考实现对比**——完整阅读参考实现，不要略读
3. **识别差异**——列出正常代码和出问题代码之间的每一个差异
4. **理解依赖关系**——这个功能需要哪些其他组件、设置、环境？

### 阶段 4 — 假设与验证

在测试任何假设之前，先生成 **3–5 个排序后的假设**。单一假设生成会锚定到第一个看似合理的想法。

每个假设必须是**可证伪的**——说明它做出的预测。

> 格式："如果 &lt;X&gt; 是原因，那么 &lt;改变 Y&gt; 会让 Bug 消失 / &lt;改变 Z&gt; 会让它更严重。"

**在测试前将排序后的列表展示给用户。** 他们通常拥有能瞬间重新排序的领域知识。如果用户不在，按你的排序继续。

每个探测必须映射到特定预测。**一次只改变一个变量。**

**为每条调试日志打上唯一前缀**，例如 `[DEBUG-a4f2]`。最后清理时只需一次 grep 就能搞定。

### 阶段 5 — 实施修复

**修复根本原因，而非症状：**

1. **创建失败的测试用例**——最简化的复现，修复前必须先有测试
2. **实施单一修复**——每次只改一处，不做"顺便改改"的优化
3. **验证修复**——测试现在通过了吗？其他测试没有被破坏吧？
4. **如果修复不起作用**——少于 3 次：回到阶段 2 重新分析；**3 次及以上：停下来质疑架构**

**3 次以上修复都失败了，表明存在架构问题：**
- 每次修复都暴露出新问题
- 修复需要"大规模重构"才能实现
- 每次修复都在其他地方产生新症状

停下来，和在阶段 4 的修复之后，**必须添加纵深防御**（参见 `defense-in-depth.md`）——在数据经过的每一层都做校验，让这个 bug 在结构上不可能发生。

### 阶段 6 — 清理 + 事后分析

在宣布完成前必须：

- [ ] 原始复现不再复现
- [ ] 回归测试通过
- [ ] 所有 `[DEBUG-...]` 探测已移除
- [ ] 临时原型已删除
- [ ] 最终被证实正确的假设已写入提交/PR 消息中

**然后问：什么能预防这个 Bug？** 如果答案涉及架构变更，将具体信息移交给 `improve-codebase-architecture`。

## 红线——停下来，按流程走

如果发现自己在想：
- "先临时修一下，以后再排查"
- "试着改改 X 看看行不行"
- "一次性改多个地方，跑测试看看"
- "大概是 X 的问题，让我修一下"
- "我不完全理解，但这应该能行"
- **"再试一次修复"（已经尝试了 2 次以上）**

**以上都意味着：停下来。回到阶段 2。**

## 实际效果

- 系统化方法：15-30 分钟修复
- 随意修复方法：2-3 小时反复折腾
- 一次修复成功率：95% vs 40%
- 引入新 bug：几乎为零 vs 经常发生
