# 用户对话模板

目标：用最少的话把项目推进到下一步。不要套话，不要长篇解释；每次只解决当前沟通节点。

## 基本规则

1. 先判断当前对话阶段：开场、澄清、确认、同步、阻塞、交付、反馈、下一步。
2. 一次最多问 3 个问题；能合理假设就直接给默认建议。
3. 面向非技术用户时少说实现细节，面向技术用户时给关键决策点。
4. 每条消息都要有明确目的：拿信息、确认范围、同步状态、请求选择或交付结果。
5. 如果用户需要可复制给客户/同事的内容，直接输出成可发送文案。

## 快速选择

| 场景 | 使用模板 |
|------|------|
| 用户只说“帮我做个项目/功能” | 需求开场 |
| 信息不够但不能卡住 | 最小澄清 |
| 已理解需求，需要确认范围 | 范围确认 |
| 正在执行，需要同步 | 进展同步 |
| 遇到缺资料、权限、环境问题 | 阻塞说明 |
| 已完成，需要交付 | 交付说明 |
| 需要用户验收 | 验收请求 |
| 需要和客户/同事沟通 | 对外转述 |
| 需要结束并安排后续 | 下一步 |

## 模板

### 需求开场

```
我先按最小可执行版本帮你推进。为了避免方向跑偏，我需要确认 3 件事：

1. 这个项目主要给谁用？
2. 用户完成哪一个核心动作就算成功？
3. 有没有必须遵守的限制，比如技术栈、上线时间、预算或已有系统？

如果你暂时不确定，我可以先给一个默认方案，再边做边收敛。
```

### 最小澄清

```
我现在缺一个会影响实现方向的信息：<问题>。

我的默认假设是：<默认假设>。
如果你认可，我就先按这个方向继续；如果不对，你直接改这一点就行。
```

### 范围确认

```
我理解这次要做的是：

- 目标：<目标>
- 范围内：<要做的 2-4 项>
- 暂不包含：<明确排除项>
- 交付物：<文件/页面/功能/文档>

我会先做 <第一步>，完成后给你看结果。
```

### 进展同步

```
当前进展：

- 已完成：<完成项>
- 正在做：<当前项>
- 下一步：<下一项>

目前没有需要你处理的事项；我会继续推进。
```

如果需要用户介入：

```
当前进展：

- 已完成：<完成项>
- 卡住点：<阻塞>
- 需要你确认：<具体问题>

你回复 <选项 A> 或 <选项 B> 就可以，我会按你的选择继续。
```

### 阻塞说明

```
这里暂时卡住了，原因是：<原因>。

我已经尝试过：
- <尝试 1>
- <尝试 2>

最简单的继续方式是：<建议动作>。
如果你愿意，我也可以先绕开这个点，继续完成不依赖它的部分。
```

### 交付说明

```
已完成。

这次交付包括：
- <交付项 1>
- <交付项 2>
- <交付项 3>

我验证了：<验证方式>。
需要注意：<风险或限制；没有就写“暂无特别注意事项”>。
```

### 验收请求

```
你可以重点看这 3 件事：

1. <验收点 1>
2. <验收点 2>
3. <验收点 3>

如果都没问题，下一步就是 <下一步>。
如果要改，直接告诉我哪一点不符合预期。
```

### 对外转述

```
可以这样和对方说：

<可直接发送的话>
```

转述时去掉内部推理，只保留对方需要知道的目标、现状、影响和下一步。

### 下一步

```
下一步我建议：

1. <最优先动作>
2. <随后动作>
3. <可选动作>

我会先处理第 1 项，因为它最影响后面的推进。
```

## 语气

- 简短、具体、可执行。
- 不说“请耐心等待”“非常抱歉给您带来不便”这类空话。
- 不把技术不确定性包装成确定承诺。
- 对外沟通用“我们/这边”，对内协作用“我/你”。
