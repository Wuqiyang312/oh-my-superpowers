# UI 原型

在单个路由上生成**几种截然不同的 UI 变体**，可通过浮动底部栏切换。用户在浏览器中翻转变体，选择一种（或从每种中借鉴部分），然后丢弃其余的。

如果问题是关于逻辑/状态而非外观——选错了分支。请使用 [LOGIC.md](LOGIC.md)。

## 何时使用该模式

- "这个页面应该长什么样？"
- "我想在决定之前看看这个仪表盘的几个选项。"
- "给设置屏幕试一个不同的布局。"
- 任何用户否则会花一天时间在脑海中挑选三个模糊原型的场景。

## 两个子模式 —— 强烈优先使用子模式 A

当 UI 原型**紧挨着应用的其余部分**时——真实的 header、真实的侧边栏、真实的数据、真实的密度——更容易评判。一个孤立的可丢弃路由是真空：每个变体单独看都不错。只要有一个合理的现有页面可以承载这些变体，就默认使用子模式 A。只有当原型真的没有附近的家时才使用子模式 B。

### 子模式 A —— 调整现有页面（优先）

路由已经存在。变体在**同一路由上**渲染，通过 `?variant=` URL 查询参数控制。现有的数据获取、参数和认证都保持不变——只有渲染部分会切换。这是默认选项；除非有特定理由，否则选择它。

如果原型针对的是还没有页面但*自然会生活在某个页面内部*的东西（仪表盘的新区域、设置屏幕的新卡片、现有流程的新步骤）——那仍然是子模式 A。将变体挂载到宿主页面内部。

### 子模式 B —— 新页面（最后手段）

仅当被原型化的东西确实没有现有页面可以生活在其中时——例如一个全新的顶层表面，或一个无法嵌入任何合理位置的流程。

创建一个**可丢弃路由**，遵循项目已有的路由约定——不要发明新的顶层结构。命名要让它明显是原型（例如在路径或文件名中包含 `prototype` 一词）。使用相同的 `?variant=` 模式。

在确定使用子模式 B 之前，先理智检查一下：真的没有现有页面可以嵌入吗？空路由会隐藏一个已填充页面会暴露的设计问题。

在两种子模式中，浮动底部栏是相同的。

## 流程

### 1. 明确问题并确定变体数量

默认使用 **3 个变体**。超过 5 个就不再是截然不同而是噪音了——上限为 5。

在原型的位置或文件顶部注释中，用一行写下计划：

> "设置页面的三个变体，通过 `?variant=` 在现有 `/settings` 路由上切换。"

这无论用户是在旁边准备提出反对还是不在，都适用。

### 2. 生成截然不同的变体

为每个变体起草。对每个变体遵循：

- 页面的目的及其可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、plain CSS，无论什么）。
- 清晰的导出组件名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须是**结构不同的**——不同的布局、不同的信息层次、不同的主要交互方式，而不仅仅是不同的颜色。三个稍作调整的卡片网格不是 UI 原型，是墙纸。如果两个草案过于相似，用明确的"不要使用卡片网格"指导重新做其中一个。

### 3. 将它们连接起来

在路由上创建一个切换组件：

```tsx
// 伪代码 —— 适配项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子模式 A（现有页面）：保留切换器上方的所有现有数据获取；只有渲染的子树随变体变化。

对于子模式 B（新页面）：`/prototype/<名称>` 下的可丢弃路由挂载相同的切换器。

### 4. 构建浮动切换器

屏幕底部中央的小型固定位置栏，包含三部分：

- **左箭头** — 循环到上一个变体（循环 wrapping）。
- **变体标签** — 显示当前变体键，如果变体导出了名称，也显示该名称。例如 `B —— 侧边栏布局`。
- **右箭头** — 向前循环（循环 wrapping）。

行为：

- 点击箭头更新 URL 查询参数（使用框架的路由器——Next 上用 `router.replace`，React Router 上用 `navigate` 等），以便变体可分享且刷新后保持稳定。
- 键盘：`←` 和 `→` 箭头键也可以循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时，不要拦截箭头键。
- 视觉上与页面区分开（例如高对比度 pill、微妙的阴影），以便它明显不是正在被评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等效检查进行控制，这样偶然合并的原型不会将栏发布给用户。

将切换器放在单个共享组件中，以便两种子模式都可以复用。将它放在项目中共享 UI 所在的位置。

### 5. 交付

展示 URL（和 `?variant=` 键）。用户会在方便时翻阅。有趣的反馈通常是**"我想要 B 的 header 配上 C 的侧边栏"**——那其实就是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，写下是哪一个以及为什么（提交信息、ADR、issue，或如果离线运行且用户尚未回应，则在原型旁边放一个 `NOTES.md`）。然后：

- **子模式 A** —— 删除失败的变体和切换器；将胜出者折叠到现有页面中。
- **子模式 B** —— 将胜出的变体提升为真实路由，删除可丢弃路由和切换器。

不要让变体组件或切换器四处散落。它们腐烂得很快，会让下一个阅读者困惑。

## 反模式

- **仅在颜色或文案上不同的变体。** 那是微调，不是原型。真正的变体在结构上有分歧。
- **变体之间共享过多代码。** 共享的 `<Header>` 没问题；共享的 `<Layout>` 就违背了目的。每个变体应该可以自由地抛弃布局。
- **将变体接入真实变更。** 只读原型没问题。如果变体需要变更，指向一个 stub——问题是"这个应该长什么样"，而不是"后端是否工作"。
- **直接将原型投入生产。** 变体代码是在原型约束下编写的（没有测试、最少的错误处理）。折叠时要正确重写。
