# 逻辑原型

一个微小的交互式终端应用，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形状**时使用——那些在纸面上看起来合理，但只有经过真实案例验证后才会感觉不对劲的东西。

## 何时使用该模式

- "我不确定这个状态机是否处理了 X 然后 Y 的边界情况。"
- "这个数据模型是否真的能表示…的情况？"
- "我想在写代码之前先感受一下 API 应该长什么样。"
- 任何用户想要**按下按钮并观察状态变化**的场景。

如果问题是"这个应该长什么样"——选错了分支。请使用 [UI.md](UI.md)。

## 流程

### 1. 明确问题

在写代码之前，先写下你要原型化的状态模型和问题是什么。一个段落即可，可以写在原型的 README 中或文件顶部的注释里。一个回答错误问题的逻辑原型完全是浪费——让问题明确化，以便后续可以检查，无论用户现在是否在关注，还是之后离线回看。

### 2. 选择语言

使用宿主项目的语言。如果项目没有明显的运行时（例如文档仓库），请询问。

遵循项目现有的工具约定——不要仅仅为了原型就添加新的包管理器或运行时。

### 3. 将逻辑隔离在可移植模块中

将实际的逻辑——回答问题的核心部分——放在一个小型的、纯的接口后面，以便日后可以直接提取并放入真实代码库。围绕它的 TUI 是可丢弃的；逻辑模块不应如此。

正确的形状取决于问题：

- **纯 reducer** — `(state, action) => state`。适用于动作是离散事件且状态是单一值的情况。
- **状态机** — 显式的状态和转换。适用于"当前哪些动作是合法的"本身就是问题的一部分时。
- **一组作用于纯数据类型的小型纯函数**。适用于没有隐式当前状态——只有转换的情况。
- **具有清晰方法表面的类或模块**，当逻辑确实拥有持续的内部状态时。

选择最符合所提问题的形状，*而不是*最容易接入 TUI 的那个。保持纯：没有 I/O，没有终端代码，没有用于控制流的 `console.log`。TUI 导入它并调用它；不会有反向的数据流。

这就是原型在其生命周期之后仍然有用的原因。当问题得到回答后，经过验证的 reducer / 状态机 / 函数集可以被提取到真实模块中——TUI 外壳则被删除。

### 4. 构建最小的状态展示 TUI

将其构建为**轻量级 TUI** —— 在每个 tick，清空屏幕（`console.clear()` / `print("\033[2J\033[H")` / 等效方式）并重新渲染整个帧。用户应该始终看到一个稳定的视图，而不是不断增长的滚动记录。

每帧有两个部分，顺序如下：

1. **当前状态**，美观打印且 diff 友好（每行一个字段，或格式化的 JSON）。对字段名或节标题使用**粗体**，对不太重要的上下文（时间戳、ID、派生值）使用**暗淡**。原生 ANSI 转义码即可——`\x1b[1m` 粗体，`\x1b[2m` 暗淡，`\x1b[0m` 重置。除非项目中已有样式库，否则不需要引入新的。
2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 时钟 tick  [q] 退出`。快捷键加粗，描述暗淡，或反之——怎么清晰怎么来。

行为：

1. **初始化状态** — 一个单一内存中的对象/结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行输入）**，分发给处理程序来修改状态。
3. **每次动作后重新渲染**完整帧——不要追加，而是替换。
4. **循环直到退出。**

整个帧应该能容纳在一个屏幕上。

### 5. 让它可以一键运行

在项目的现有任务运行器中添加一个脚本（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应该运行 `pnpm run <原型名称>` 或等效命令——不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

### 6. 交付

给用户运行命令。他们会自己驱动它；有趣的瞬间是他们说"等等，那不应该可能发生"或"嗯，我原以为 X 会不同"的时候——这些就是*想法*中的缺陷，而这正是重点所在。如果他们想添加新的动作，就添加它们。原型是演进的。

### 7. 捕获答案

当原型完成它的使命后，问题的答案是唯一值得保留的东西。如果用户在旁边，询问它教会了他们什么。如果不在，在原型旁边留下一个 `NOTES.md`，以便在删除原型之前可以填写答案（或者由你，如果你观察了会话过程，来填写）。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要接入真实数据库。** 除非问题明确涉及持久化，否则使用内存存储。
- **不要泛化。** 不要"如果我们以后想支持 X 怎么办"。原型只回答一个问题。
- **不要将逻辑和 TUI 混在一起。** 如果 reducer / 状态机引用了 `console.log`、prompts 或终端转义码，它就不再是可移植的。将 TUI 保持为覆盖在纯模块之上的薄壳。
- **不要将 TUI 外壳投入生产。** 外壳针对从终端手动驱动进行了优化。它背后的逻辑模块才是值得保留的部分。
