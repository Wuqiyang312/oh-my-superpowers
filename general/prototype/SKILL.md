# 原型

原型是**回答问题的可丢弃代码**。问题决定形状。

## 选择分支

识别正在回答的问题——从用户的提示、周围代码中识别，或者如果用户在旁边就直接问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个微小的交互式终端应用，将状态机推过那些在纸面上难以推理的案例。
- **"这个应该长什么样？"** → [UI.md](UI.md)。在单个路由上生成几种截然不同的 UI 变体，可通过 URL 查询参数和浮动底部栏切换。

两个分支产生的产物截然不同——选错了会浪费整个原型。如果问题确实模糊且用户无法联系，默认选择更匹配周围代码的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部声明假设。

## 共同适用的规则

1. **从一开始就是可丢弃的，并且明确标记。** 将原型代码放在它实际将被使用的位置附近（紧挨着它正在原型化的模块或页面），这样上下文显而易见——但命名要让随便一看的读者知道它是原型，不是生产代码。对于可丢弃的 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一条命令运行。** 无论项目现有任务运行器支持什么——`pnpm <名称>`、`python <路径>`、`bun <路径>` 等。用户必须能够不假思索地启动它。
3. **默认不持久化。** 状态驻留在内存中。持久化是原型正在*检查*的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，请访问一个暂存数据库或带有明确"原型 — 可删除我"名称的本地文件。
4. **跳过打磨。** 没有测试，没有超出使原型*可运行*所需的错误处理，没有抽象。重点是快速学习一些东西，然后删除它。
5. **展示状态。** 每次动作（逻辑）或每次变体切换（UI）后，打印或渲染完整的关联状态，以便用户可以看到什么发生了变化。
6. **完成后删除或吸收。** 当原型回答了它的问题后，要么删除它，要么将经过验证的决定折叠到真实代码中——不要让它在仓库中腐烂。

## 完成后

原型的*答案*是唯一值得保留的东西。将它捕获在某个持久的地方（提交信息、ADR、issue，或原型旁边的 `NOTES.md`），连同它正在回答的问题一起。如果用户在旁边，这个捕获就是一次简短的对话；如果不在，就留下占位符，让他们（或你，在下一次遍历时）可以在删除原型之前填写结论。
