# CONTEXT.md 格式

## 结构

```md
# {上下文名称}

{一两句话描述这个上下文是什么以及为什么存在。}

## 语言

**订单（Order）**：
{对该术语的简明描述}
_避免使用_：Purchase, transaction

**发票（Invoice）**：
交付后向客户发出的付款请求。
_避免使用_：Bill, payment request

**客户（Customer）**：
下订单的个人或组织。
_避免使用_：Client, buyer, account

## 关系

- 一个 **订单（Order）** 产生一个或多个 **发票（Invoice）**
- 一个 **发票（Invoice）** 恰好属于一个 **客户（Customer）**

## 示例对话

> **开发者：**"当一个 **客户（Customer）** 下了一个 **订单（Order）** 时，我们立即创建 **发票（Invoice）** 吗？"
> **领域专家：**"不——只有当 **履约（Fulfillment）** 确认后才会生成 **发票（Invoice）**。"

## 标记的歧义

- "account" 被同时用来表示 **客户（Customer）** 和 **用户（User）**——已解决：这是两个不同的概念。
```

## 规则

- **要有主见。** 当同一概念存在多个词汇时，选择最好的一个，将其他列为应避免使用的别名。
- **显式标记冲突。** 如果一个术语被模糊使用，在"标记的歧义"中明确指出，并给出清晰的解决方案。
- **定义要精简。** 最多一句话。定义它*是*什么，而不是它*做*什么。
- **展示关系。** 使用粗体术语名称，并在明显时表达基数。
- **只包含特定于本项目上下文的术语。** 通用编程概念（超时、错误类型、工具模式）即使项目广泛使用也不应包含。在添加术语前，问：这是该上下文独有的概念，还是一个通用编程概念？只有前者才属于这里。
- **当自然集群出现时，在子标题下对术语分组。** 如果所有术语都属于一个内聚的领域，平铺列表也可以。
- **编写示例对话。** 开发者与领域专家之间的对话，展示术语如何自然地交互，并澄清相关概念之间的边界。

## 单上下文 vs 多上下文仓库

**单上下文（大多数仓库）：** 仓库根目录下有一个 `CONTEXT.md`。

**多上下文：** 仓库根目录下有一个 `CONTEXT-MAP.md`，列出各个上下文、它们的位置以及彼此之间的关系：

```md
# 上下文地图

## 上下文

- [Ordering](./src/ordering/CONTEXT.md) — 接收和跟踪客户订单
- [Billing](./src/billing/CONTEXT.md) — 生成发票并处理付款
- [Fulfillment](./src/fulfillment/CONTEXT.md) — 管理仓库拣货和发货

## 关系

- **Ordering → Fulfillment**：Ordering 发出 `OrderPlaced` 事件；Fulfillment 消费它们以开始拣货
- **Fulfillment → Billing**：Fulfillment 发出 `ShipmentDispatched` 事件；Billing 消费它们以生成发票
- **Ordering ↔ Billing**：共享 `CustomerId` 和 `Money` 类型
```

该技能会推断适用哪种结构：

- 如果存在 `CONTEXT-MAP.md`，读取它以找到上下文
- 如果只有根目录下的 `CONTEXT.md`，则为单上下文
- 如果两者都不存在，在第一个术语被解析时惰性创建根目录下的 `CONTEXT.md`

当存在多个上下文时，推断当前主题与哪个相关。如果不清楚，询问。
