# ADR 格式

ADR 存放在 `docs/adr/` 目录下，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

惰性创建 `docs/adr/` 目录——只在需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么，我们决定了什么，以及为什么。}
```

就这些。一个 ADR 可以只有一段话。其价值在于记录*做出了*某个决策以及*为什么*——而不是填满各个章节。

## 可选章节

只在它们真正有价值时才包含。大多数 ADR 不需要它们。

- **Status** 前置元数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——在决策被重新审阅时有用
- **Considered Options**——只有当被拒绝的替代方案值得记住时才包含
- **Consequences**——只有当需要指出非显而易见的下游影响时才包含

## 编号

扫描 `docs/adr/` 目录下已有的最高编号并加一。

## 何时提议 ADR

以下三个条件必须全部满足：

1. **难以逆转**——日后改变主意的成本是实质性的
2. **没有上下文会显得意外**——未来的读者看到代码会想"他们到底为什么这么做？"
3. **真正权衡的结果**——存在真实的替代方案，你出于特定原因选择了其中一个

如果一个决策容易逆转，跳过它——你直接改就是了。如果它不令人意外，没人会问为什么。如果没有真正的替代方案，除了"我们做了显而易见的事"之外没有什么可记录的。

### 什么情况下值得写

- **架构形态。** "我们使用单体仓库。" "写入模型是事件溯源的，读取模型被投影到 Postgres 中。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而非同步 HTTP。"
- **会带来锁定效应的技术选型。** 数据库、消息总线、认证提供商、部署目标。不是每个库——只写那些需要一季度时间才能替换掉的。
- **边界和范围决策。** "Customer 数据由 Customer 上下文拥有；其他上下文只通过 ID 引用它。" 明确的否定与肯定同样有价值。
- **有意偏离显而易见的路线。** "我们使用手动 SQL 而不是 ORM，因为 X。" 任何合理读者会假设相反方案的地方。这能阻止下一个工程师"修复"一个实际上是故意的选择。
- **代码中不可见的约束。** "由于合规要求我们无法使用 AWS。" "由于合作伙伴 API 合同，响应时间必须低于 200ms。"
- **拒绝非显而易见时的被拒绝替代方案。** 如果你考虑过 GraphQL 并出于微妙原因选择了 REST，记录下来——否则六个月后有人会再次提议 GraphQL。
