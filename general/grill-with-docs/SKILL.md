<caveman>

grill 期间：用压缩格式提问和记录。砍掉废话、填充词、客套话。保留技术精确性。减少 token 消耗 -> 降低历史压缩丢失关键决策的风险。

具体：简短提问，直接记录术语/决策到 AGENTS.md/docs/agents/ADR，不使用"Could you help me understand..."之类客套。

</caveman>

<what-to-do>

就这个方案的每个方面毫不留情地追问，直到我们达成共同理解。沿着设计树的每个分支逐一排查，逐个解决决策之间的依赖关系。对于每个问题，提供你推荐的答案。

一次只问一个问题，在每个问题得到反馈后再继续。

如果某个问题可以通过探索代码库来回答，那就去探索代码库。

</what-to-do>

## 领域感知

在探索代码库期间，同时查找现有文档：

### 文件结构

大多数仓库只有一个上下文：

```
/
├── AGENTS.md
├── docs/
│   ├── agents/
│   │   ├── issue-tracker.md
│   │   ├── domain.md
│   │   └── toolchain.md
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果根目录下存在 `CONTEXT-MAP.md`，则仓库有多个上下文。地图指出每个上下文所在的位置：

```
/
├── AGENTS.md
├── docs/
│   ├── agents/                       ← 全系统配置
│   └── adr/                          ← 全系统范围的决策
├── src/
│   ├── ordering/
│   │   ├── AGENTS.md
│   │   └── docs/adr/                 ← 上下文特定的决策
│   └── billing/
│       ├── AGENTS.md
│       └── docs/adr/
```

惰性创建文件——只在有内容可写时才创建。如果不存在 `AGENTS.md`，在第一个术语被解析时创建。如果不存在 `docs/agents/`，在需要第一个配置文件时创建。如果不存在 `docs/adr/`，在需要第一个 ADR 时创建。

## 会话期间

### 对照术语表进行挑战

当用户使用的术语与 `AGENTS.md` 中现有语言冲突时，立即指出。"你的术语表将 'cancellation' 定义为 X，但你似乎指的是 Y——到底是哪个？"

### 打磨模糊语言

当用户使用模糊或重载的术语时，提议一个精确的规范术语。"你说的是 'account'——你指的是 Customer 还是 User？它们是不同的东西。"

### 讨论具体场景

在讨论领域关系时，用具体场景对它们进行压力测试。发明探测边界情况的场景，迫使用户对概念之间的边界做出精确说明。

### 与代码交叉验证

当用户说明某物如何工作时，检查代码是否一致。如果你发现矛盾，提出来："你的代码取消了整个 Order，但你刚说部分取消是可能的——哪个是对的？"

### 即时更新 AGENTS.md

当一个术语被解析时，立即更新 `AGENTS.md`。不要批量处理——发生时就捕获。使用 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) 中的格式。

`AGENTS.md` 应该完全没有实现细节。不要把 `AGENTS.md` 当作规格说明书、草稿纸或实现决策的仓库。它只是一个术语表，仅此而已。

### 输出设计文档

grilling session 结束后，产出一份 `docs/design/<feature>.md` 设计文档，作为与 `writing-plans` 的交接件：

```markdown
# <功能名称> 设计文档

## 目标
一句话描述要构建什么、解决什么问题

## 范围
- 包含：具体做什么
- 排除：明确不做什么（留给 v2）

## 验收标准
- 作为 <角色>，我可以 <做什么>，以便 <价值>

## 边界情况
- 系统处于状态 Z 时收到请求 → 怎么处理？
- 第三方服务超时 → 降级策略？

## 关键设计决策
- 为什么选 A 而不是 B

## 相关文档
- 引用的 AGENTS.md 术语
- 相关的 ADR
```

**模式选择：** 设计文档输出后，提供两种路径：
- **模式 1：** 仅设计 → "下一步使用 `writing-plans` 从 `docs/design/<feature>.md` 生成执行计划"
- **模式 2：** 设计 → 计划 → 执行 → "下一步使用 `writing-plans` 生成计划，再用 `executing-plans` 并行执行"

### 审慎提议 ADR

只有在三个条件都满足时才提议创建 ADR：

1. **难以逆转**——日后改变主意的成本是实质性的
2. **没有上下文会显得意外**——未来的读者会想"他们为什么这么做？"
3. **真正权衡的结果**——存在真实的替代方案，你出于特定原因选择了其中一个

如果三个条件中缺少任何一个，跳过 ADR。使用 [ADR-FORMAT.md](./ADR-FORMAT.md) 中的格式。
