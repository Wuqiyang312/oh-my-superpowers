# 接口设计

当用户想要为选定的深化候选探索替代接口时，使用此并行子 agent 模式。基于 Ousterhout 的"Design It Twice"——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 界定问题空间

在生成子 agent 之前，为选定的候选写一个面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖什么，以及它们属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图来使约束具体化——不是提议，只是让约束具体化的一种方式

展示给用户，然后立即进入步骤 2。用户在阅读思考的同时，子 agent 可以并行工作。

### 2. 生成子 agent

使用 Agent 工具并行生成 3 个以上的子 agent。每个必须产生一个**截然不同**的深模块接口。

为每个子 agent 提供独立的技术简报（文件路径、耦合细节、[DEEPENING.md](DEEPENING.md) 中的依赖类别、接缝背后是什么）。该简报独立于步骤 1 中面向用户的问题空间说明。给每个 agent 一个不同的设计约束：

- Agent 1："最小化接口——最多 1–3 个入口点。最大化每个入口点的杠杆效应。"
- Agent 2："最大化灵活性——支持多种用例和扩展。"
- Agent 3："为最常见的调用方优化——让默认情况变得轻而易举。"
- Agent 4（如适用）："围绕端口与适配器设计跨接缝依赖。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，以便每个子 agent 的命名与架构语言和项目领域语言保持一致。

每个子 agent 输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 展示调用方如何使用它的使用示例
3. 实现隐藏在接缝背后的是什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆效应高，哪里薄弱

### 3. 展示与比较

顺序展示设计以便用户能吸收每一个，然后用文字进行比较。从**深度**（接口处的杠杆效应）、**局部性**（变化集中在哪里）和**接缝位置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果来自不同设计的元素可以很好地组合，提议一个混合方案。要有主见——用户想要一个有力的观点，而不是一个菜单。
