# 深化

如何在给定依赖的情况下，安全地将一簇浅层模块深化。假设已熟悉 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**、**interface**、**seam**、**adapter**。

## 依赖分类

在评估一个深化候选时，对其依赖进行分类。类别决定了深化后的模块如何在其接缝处进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。总是可深化——合并模块并通过新接口直接测试。不需要适配器。

### 2. 本地可替代（Local-substitutable）

拥有本地测试替代品的依赖（Postgres 的 PGLite、内存文件系统）。如果替代品存在则可深化。深化后的模块在测试套件中使用替代品运行。接缝是内部的；模块外部接口处没有端口。

### 3. 远程但自有（端口与适配器）

你自己的服务跨越网络边界（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有逻辑；传输被注入为**适配器**。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

推荐形式：*"在接缝处定义一个端口，为生产环境实现 HTTP 适配器，为测试实现内存适配器，这样即使逻辑部署在网络两端，也能 sitting in one deep module。"*

### 4. 真正的外部（Mock）

第三方服务（Stripe、Twilio 等）你无法控制。深模块将外部依赖作为注入端口接收；测试提供模拟适配器。

## 接缝纪律

- **一个适配器意味着假设的接缝。两个适配器意味着真正的接缝。** 除非至少有两个适配器是合理的（通常是生产环境 + 测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深模块可以有内部接缝（对其实现私有，由其自身测试使用）以及其接口处的外部接缝。不要因为测试使用了内部接缝就通过接口暴露它们。

## 测试策略：替换，不要叠加

- 浅层模块上的旧单元测试一旦深模块接口处存在测试，就变成了垃圾——删除它们。
- 在深模块的接口处编写新测试。**接口就是测试表面。**
- 测试通过接口断言可观察的结果，而非内部状态。
- 测试应该能经受内部重构——它们描述行为，而非实现。如果实现改变时测试必须改变，那就是测试越过了接口。
