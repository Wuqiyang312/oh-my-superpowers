# 术语

本技能所有建议的共享词汇。请精确使用以下术语——不要用"组件"、"服务"、"API"或"边界"来替代。术语一致性本身就是重点。

## 概念

**模块（Module）**
任何具有接口和实现的东西。刻意不区分尺度——它同样适用于函数、类、包，或跨层级的切片。
_避免使用_：单元、组件、服务。

**接口（Interface）**
调用者正确使用模块所需了解的一切。包括类型签名，还包括不变量、顺序约束、错误模式、所需配置和性能特征。
_避免使用_：API、签名（过于狭隘——这些仅指类型层面的表面）。

**实现（Implementation）**
模块内部的内容——它的代码体。与**适配器（Adapter）**不同：一个东西可以是小型适配器配合大型实现（如 Postgres 仓库），也可以是大型适配器配合小型实现（如内存 fake）。当讨论的重点是"缝合处"时用"适配器"；其他情况用"实现"。

**深度（Depth）**
接口处的杠杆效应——调用者（或测试）每学习一个单位的接口，就能调用的行为量。当一个模块在小型接口背后封装了大量行为时，它就是**深的**。当接口几乎和实现一样复杂时，它就是**浅的**。

**缝合处（Seam）**_（来自 Michael Feathers）_
一种可以在不编辑该位置的情况下改变行为的地方。模块接口所在的*位置*。选择缝合处的位置本身就是一个设计决策，与接口背后的内容无关。
_避免使用_：边界（与 DDD 的有界上下文概念重叠）。

**适配器（Adapter）**
在缝合处满足接口的具体事物。描述的是*角色*（它填充什么槽位），而非实质（内部是什么）。

**杠杆效应（Leverage）**
调用者从深度中获得的东西。每学习一个单位的接口，获得更多的能力。一个实现在 N 个调用点和 M 个测试中都能复用。

**局部性（Locality）**
维护者从深度中获得的东西。变更、缺陷、知识和验证集中于一处，而非扩散到各调用者。修复一次，处处生效。

## 原则

- **深度是接口的属性，不是实现的属性。** 一个深的模块在内部可以由小的、可 mock 的、可替换的部分组成——只是它们不是接口的一部分。模块可以有**内部缝合处**（对实现私有，仅供其自身测试使用），也可以有位于接口处的**外部缝合处**。
- **删除测试。** 想象删除这个模块。如果复杂度消失了，说明这个模块没有隐藏任何东西（它只是个透传）。如果复杂度在 N 个调用者中重新出现，说明这个模块确实有其存在的价值。
- **接口就是测试表面。** 调用者和测试跨越同一个缝合处。如果你想测试接口*背后*的东西，那说明模块的形状可能有问题。
- **一个适配器意味着假设性的缝合处。两个适配器意味着真实的缝合处。** 除非缝合处两侧确实有变化，否则不要引入缝合处。

## 关系

- 一个**模块**有且仅有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**的属性，相对于其**接口**来衡量。
- **缝合处**是**模块**的**接口**所在的位置。
- **适配器**位于**缝合处**并满足**接口**。
- **深度**为调用者产生**杠杆效应**，为维护者产生**局部性**。

## 被拒绝的框架

- **将深度视为实现行数与接口行数的比率**（Ousterhout）：这会鼓励填充实现。我们使用"深度即杠杆效应"的定义。
- **将"接口"等同于 TypeScript 的 `interface` 关键字或类的公共方法**：过于狭隘——此处的接口包括调用者必须了解的每一个事实。
- **"边界"**：与 DDD 的有界上下文概念重叠。请使用**缝合处**或**接口**。
